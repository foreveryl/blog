Naive Bayes

数学推导

Naive Bayes 其实质是生成时模型.
所谓生成时模型, 其实是分别针对正, 负 样本, 研究其中data 中各个维度的分布情况.

推导过程:

  • 目标
    希望能能从给定的样本点 $x$ 得出他在$y$ 上的分布. 如二分类问题, 就是判断x 是正还是负样本. 即:
    $p(y | x )$

  • 贝叶斯转换:
    不同于LR, NB是生成时模型，体现在:
    $p(y | x ) = \frac {p(x | y ) * p(y) } {p(x,y) } $

在这个公式里, 分母$p(x,y)$ 是常量. 不做考虑.
分子
$p(y)$ 是$y$ 分布的概率. 已二分类为例, 那就是正负样本的比例. 给定样本, 就容易计算. 这各不提.
$p(x | y) $ 这个就是生成时模型的本质:
每一个y值下, 研究x 的分布情况.

为什么叫Naive呢?
这里是有一个假设, 就是$x$ 向量中每一个维度值独立分布的. 其实 LR 也是一样.

于是有推导:
$p(y | x) => p(x | y ) p(y) $
$ = p(x_{1} | y ) p (x{2} | y) … p (x{n} | y ) * p(y) $

为减小运算. 这里需要乘法转加法
$p(y | x ) = log p(x{1} | y ) + log p(x{w} | y ) + … + log p(x_{n} | y ) + log p(y) $

也就是对于二分分类,
计算:
$p(y=1 | x ) = log p(x{1} | y=1 ) + log p(x{2} | y=1 ) + … + log p(x{n} | y=1 ) + log p(y=+1) $
$p(y=0 | x ) = log p(x{1} | y=0) + log p(x{2} | y=0 ) + … + log p(x{n} | y=0 ) + log p(y=0) $
然后看那个概率大, 从而分类.

现在的问题是, 如何求 $p(x | y=+1 )$ , $p(x | y=0 )$

$p(x | y)$ 求解, 这个的关键是看, x 是什么变量.
  • 如果 x 是 0, 1 这样的离散变量. 即是否出现.
    也就是假设 x 服从Bernoulli 分布. 也叫[0,1] 分布.
    $ P(x) = p$ when $n = 1 $
    $ P(x) = 1-p when $x = 0 $
    连续n个实验(即 n 个样本) , 就叫2项分布.
    并且由统计知识:
    $E(n) = np$ 解释: n个样本中, 有正样本的个数.
    这时候我们就可以从样本点中, 估算出$p$值的大小.
    做法:
    一堆正样本($m$个) 中, 有x 这个feature的样本的比例${n}$ , 于是:
    $ p(x | y = 1) = \frac {n + \alpha} {m + \alpha * class_number } $
    加 $\alpha$ 是做拉普拉斯平滑

由于这种x 是满足Bernoulli分布. 这种也叫Bernoulli Naive Bayes

  • 如果 x 是可数的多个离散变量
    比如一个筛子可以有6个面.
    即假设x是服从Multinomial 分布. 这个就是对上面Bernoulli 分布的一个扩展.
    其实算法是和上面是一样的.
    $r$ 维的随机变量$$ 服从多项式分布$M(n, p_1, p_2, … , p_r)$
    那么对期望:
    $E(k_1, k_2, …, k_r) = ( n p_1, n p_2, … , n * p_r ) $
    扫一遍数据, 就可以求出 $p_1, p_2, … , p_r$ 这些值.

这里的x 服从Multinomial分布, 所以这个叫Multinomial NB

  • 如果 x 是连续变量.
    这时候需要做假设, 假设x 是服从正太分布.
    那么从正, 负样本的观测值中, 可分别预估出各自的正太分布函数. 即 正太分布, 各自方差和期望的计算.
    那么, 对于任一个input, 就能计算出其概率的分布.

因为假设x 是服从正太分布的, 所以这种叫: Gaussian NB

  • 总结
    对于变量, 假设其服从某个分布. 然后各在正, 负样本的中x的取值, 来求出该分布的参数. 即为: $p(x | y)$
    具体求参数的方法, 需要依据实际的数据中你对样本分布的假设.

End

可见, feature 不行, 运用的算法也不同.
我猜, 这个肯定是一个算法有时好用, 有时不好用的一个原因吧.
处理的数据不一样 :)

word2vector 理解

By yl

发表于 2015-07-27

引入:

尽管word2vec是从神经概率语言模型引入的. 但是和神经网络关系其实并不大. 其实质的目标函数是: (已cbow为例):
max p(w | context(w) ) 这个最大似然函数.

context(w)是什么?

这里就有两种解法:

  • hierachical softmax
    通过词频构建了一个huffman tree. (假设字典中有D个词, 那么肯定是有D-1个非叶子节点和D个叶子节点. )
    然后从根检点到 w 的路径上的中间节点, 认为就是w 的context(w)节点.
    然后做了个假设: 如果路径节点d 到w 是向左子树进行, 那么就是正样本. 向右就是负样本.

  • negative sampling:
    这个对负样本的构建更简单了, 简单说就是sampling.
    这个过程中对高频词有做降权处理.

loss 函数

现在问题转化为:
对word w, 其vector定义为:　sum( 前后词的向量和), 找到正负样本 d.
定义loss 函数:
min(1 - sigmod( dot(w, d) ))

这样就变成典型的矩阵(word—word)矩阵分解的问题了.
通过随机梯度下降. 可以求解.

具体正负样本找的方法:
以hierachical softmax为例: 就是huffman树上从word节点到根节点path上的非叶子节点.
并且认为定义: 左子树为正, 右子树为负.

整体训练过程总结

以cbow + hierachical softmax为例

  • huffman tree 构建.
    没错, 就是数据结构课本上做编码压缩的那个.

为什么要用huffman tree?
其实不为什么, 作者原文里解释, 因为效果好……
不过他确实work. 后续可以考虑相同思路.

  • word 选择
    内存优化原因, 低频word 被过滤

  • 训练开始
    1), 模型初始化. word 默认vector
    包括huffman tree上非叶子节点.

2), 多个线程, 并行开始迭代.
a, 从文本中拿到word. sum 相邻word 的vector 为该word的vector
b, 获取正负样本.
c, sgd
d, 更新word 的vector

反思, why it works

w2v出来的时候, 流行了一大阵. 大家都觉得效果很好. 什么深度学习等等都扯出来了.
看过上面列举的求解流程, 其实和deep learning 没有任何关系.

那为啥大家都觉得好?
个人觉得一下几个原因:

  • 每个word 定义为左右word 向量之和, 很大程度上保证了相关性.
  • 训练速度快:
    多线程 asp 方式更新模型数据. 单机处理, 不加锁, 更新很快.
  • 热门降权
    使得低频次word 不会都相似到高频word上. 有了区分性.
    个人觉得这个是比一般mf 好的原因.
  • embedding
    相比mf, cf 这种计算相似次的方法. w2v训练处一个可用的向量表示.
    可以做为input, 用于其他模型中.
    尝试做了用户query 分类. 很简单的应用, 但是效果也不错.
End

当然, word2vec 也不是万能的, 同质训练数据也很重要.
至于 向量间加减 这种操作, 个人试用下来, 效果一般.

Logistic Regression

名字里是回归, 实际是做分类的.
但其实是殊途同归的.

问题定义: max likelihood

去正样本为1, 负样本为0
对某一个样本, 在0,1 上有概率分布. (分布logistic link 函数)

$
p_{i} = \frac{1}{1 + e^{-wx} }
$

那么对分类问题, 可规定 >0.5 为正样本. <0.5就是负样本.

如何定义loss 函数 ? 很直观, 希望train set中+1的$p{i}$越接近1, 否则越接近0. 于是:
$
L=\sum{i, y{i} = 1} log(p{i} ) + \sum_{i, y{i} = 0} log(1 - p{i} )
$

显然我们需要求解的是 $ max L $

GD 方法求解

要求最大值, 用梯度上升

$w = w + \alpha * \frac { \partial {L} } {\partial {w} }$

$\alpha$ 叫learning rate, 在SGD中是人工置顶的常量(后者有衰减因子). 后面会提到line search 中步长的概念. 先不管.
于是现在的问题是, 如何求偏导, 推导过程如下:

$
\frac {\partial log p } {\partial w } = \frac {1} {p} * \frac {\partial p } {\partial w}
$
令 $k = e^{-wx}$, 根据 p 的定义: $ p_{i} = \frac{1}{1 + e^{-wx} } $

$\frac {\partial } {\partial w} ( \frac {1} { 1 + k } ) $
$= \frac { \partial } { \partial k } ( \frac {1} { 1+ k} ) \frac {\partial k } {\partial w }$
$= -1 (1 + k ) ^{-2} \frac {\partial} {\partial w } e ^ {- wx }$
$= -1 (1 + k ) ^{-2} e ^ {- wx } ( - x ) $
$= \frac {k} { ( 1 + k ) ^ 2 } x $
$= \frac {1} { 1+ k} \frac {k} { 1+ k } x$
$= p (1-p ) * x $

再回到原来的公式:
$\frac {\partial} { w} log p$
$= \frac{1}{p} \frac {\partial p } {\partial w} $
$= \frac{1} {p} p (1-p ) x $
$= (1-p) * x $

同理:
$\frac {\partial} { w} log (1-p) $
$= -1 \frac{1}{1 - p} \frac {\partial p } {\partial w} $
$= -1 \frac{1} {(1 - p) } p (1-p ) x $
$= -p x $
$ = (0 - p ) x $

对比这两个公式: 1和0 正好是样本1,0值的情况. 于是两个公式就可以总结起来为:
$ \frac { \partial {L} } {\partial w } = \sum{i} (y{i} - p{i} ) x{i} $

推导完毕.
所以一阶梯度的求解也就明白了:
$ (y - p ) $ 叫残差, 就是每个样本点的真值与预测值之间的差. $ x $ 是这个点的维度向量.

这个推导过程明确后, 求解是非常容易的.
求解梯度的过程是每个点各求残杀, 然后累加. 那么就可以并行来实现.

line search

上面的GD 方法求解. 作为优化方法的引入, 可以明确:
1), LR是怎么定义, 并且推导的.
2), 编程是如何实现的. 特别是求解1阶梯度.

回顾 $w$ 等迭代更新:
$
w = w + \alpha * \frac { \partial {L} } {\partial {w} }
$

$ \frac { \partial {L} } {\partial {w} }$ 梯度定义了方向. 但是按这个方向, 走多远是个问题.
GD 方法是人工的凭经验指定了学习率 learning rate, 也叫步长
步长太小收敛的慢, 太大则可能扯到蛋.

所以步长的选择, 就需要定义必要的条件.
有两个:

  • sufficient decrease condition
    简单说就是每次 $f(w{t+1})$ 必须小于(假设就最小值) $f(w{t})$
    更强势的条件:
  • curvature condition
    简单说, 就是要保证 $\bigtriangledown f(w{t+1})$ 要小于 $\bigtriangledown f(w{t})$
    想想为啥要小于 ?

同时满足这两个条件, 叫做wolfe conditions
当然, 如果只满足第一个条件, 也是对的. 但是收敛会慢一些.
同时看到, 梯度下降解法无法保证满足第一个条件, 所以收敛会更慢.

如果找到一个值, 不满足条件怎么办?
需要通过插值算法, 在(x, x+a) 中间找到合适的.

newton 方法

回顾newton 方法, 近似求f(x) = 0时 x 的值?

这个的切线方程如下:
$y - f(x{k}) = \bigtriangledown f(x{k}) (x - xk)$
$\Rightarrow 0 - f(x{k}) = \bigtriangledown f(x_{k}) (x - xk)$
$\Rightarrow x = x{k} - ( \bigtriangledown f(x{k}) )^{-1} * f(x{k})$

等等, 为什么要求f(x) = 0是的x呢? 我们的目标不是求loss 函数 $L$ 最小(大)值么?

依据定义. 最小值, 最大值出现的点, 1阶导数为0 .
所以, 这里的f 其实是 loss 函数 $L$ 的导函数.
所以 w 的更新是这样的:

$
w{k+1} = w{k} - ( \bigtriangledown^{2} f(w{k}) )^{-1} * \bigtriangledown f(w{k})
$

这个可以看做, 步长为1, 方向为: $( \bigtriangledown^{2} f(w{k}) )^{-1} * \bigtriangledown f(w{k})$

L-BFGS

newton法公式很完美. 可惜执行时: 需要2阶hess矩阵. 且需要求逆. 这很难办.

  • Quasi-Newton Method
    使用如下公式拟合二阶矩阵:
    $\bigtriangledown^{2} f(w{k}) \approx \frac {\bigtriangledown f(w{k+1} - \bigtriangledown f(w{k}) } { w{k+1} - w{k} }$
    所以:
    拟合hess矩阵计算方式:
    $B    {k+1} * s{k} = y{k}$
    其中:
    $s{k} = w{k+1} - w{k} $
    $y    {k} = \bigtriangledown f(w{k+1} - \bigtriangledown f(w{k}) $

  • BFGS
    拟合Hess矩阵后, 还需要求逆, 干脆直接拟合求逆后的hess 矩阵.
    $H{k+1} * y{k} = s_{k}$

  • L-BFGS
    H 矩阵是n*n 的稠密矩阵. 计算起来代价巨大.
    L-BFGS优化了这点, 只用最近几步的来拟合.

Regulation

机器学习算法有三部分:
1), loss函数.
2), optimization, 求解算法
3), regulation

前面降了loss 函数, 以及求解算法. 这个提下regulation
一般有L1, 和L2
L2 连续可求导, 比较简单. 直接加上就行.
L1 连续, 但是在0点不可导. 稍微麻烦点.
处理的逻辑是:在０点做平滑．　并且在更新时，不能超过同一象限．　

End

总结了之前实现的LR算法. 从数学定义, 到优化求解, 以及求解伪代码.
可以看到, 其实优化求解, regulation 都是通用的. 前提是, 定义的loss函数可以求导.
把logistic link 函数换乘其他的, 就可以解别的问题. 如linear regression, svm等等.

理解celery调度流程(1) Kombu消息处理

celery 是一个分布式的消息管理, 任务调度系统.
之前用户hbase scan 任务上, 非常好,并且简单的解决遇到的问题.
目前计划基于celery, 做一个能替代crontab 部署, 并且进一步对任务进行并发控制等的任务提交平台.
所以需要进一步的了解celery执行的过程.

celery 基于Kombu, 后者是一个封装AMQP的消息库.

AMQP

AMQP全称Advanced Message Queuing Protocol，是一个开放标准协议，定义一个异步消息传递所使用的应用层协议规范。
如果之前接触过zeromq, rabbitmq等对一下这些概念不会陌生

Producer: 生成着, 发布消息. 比如, 生成任务.
Broker: 收集, 管理, 发布消息
Exchange: 想象邮局, 是消息的转发机制
Queues: 消息的队列.
Consumer: 处理消息.

基于celery的hbase scan任务调度

基于celery的hbase scan任务调度

问题来源

我们开发了一个基于hbase 的多维度查询, 分析系统. 有我们的用户直接提交查询任务. 查看报表.
刚开始, 任务不多. 那么通过crontab 来每次执行一个任务. 最简单, 没问题.
但是后来用的人越来越多. 提交的查询也逐渐变多. 这样, 同时只能1个任务执行, 任务等待时间很长.
考虑到hbase 是共用资源, 并且每个任务scan 都需要和region server rpc 通信. 那么如果同时多个任务一起跑,
就会越来越慢.
所以需要并发控制.

考虑的解决方案:

  • 开发一个java 多线程任务
    一开始的思路, 写个java, 开n个线程. 每个线程run一个任务.
    缺点:
    其实看似很简单, , 看着不复杂, 但是仔细想想,还是挺麻烦
    1), 要写代码, 要debug, 要维护
    2), 多线程等于要写一个生产者, 消费者模型. 相当于自己实现一个消息队列.
    3), 如何提交任务? 还要开发一个接口
    4), 如何扩展, 要自定义协议.
    5), 如何容错？　任务运行失败能否自动重试（比如hbase 集群重启)
    消息队列被kill, 怎么恢复? 等等.
    6), 任务管理, 正在运行的, 已经运行的.. 等等.

  • 现有调度系统
    公司另外一个组, 他们开发一套任务分布式提交的系统. 并且一直在线上使用.
    优点是, 他们做到了任务的自动部署. 以及中央管理.
    调研后,发现不满足如下需求
    1), 他们的目标是 提交任务机器的并发控制, 不是任务的并发控制.
    任务的并发, 交给hadoop queue 来调度.
    机器并发是什么意思? 因为他们任务很多, 入口机内存不大, 这样在启动任务的时候, 需要找一台相对空闲的机器来执行.
    对hbase scan任务来说, 也不太适合. 我们的瓶颈是在hbase.


    2), 没有容错.
    他们的容错是通过 类似于crontab 来执行的.
    也就是这次失败, 会再下次crontab 设定的时间, 触发任务再执行.
    也不适合hbase scan 分析任务.

  • celery
    后来找到了celery. 一个python的分布式消息处理系统. 很好的解决了我们的需求.
    1), 任务实时提交
    通过前段的tornado, 任务有客户端提交执行. 不需要crontab做中间层, 在mysql中查询任务.
    2), 并发控制
    celery 的concurrence 配置就行.
    3), 容错:
    celery 支持容错. 再也不用担心jobtracker 重启了.
    4), 可靠, 可视化
    celery 是一个比较成熟的方案.
    基于celery的flower , 可以监控任务的执行. celery的状态. 等等.
    5), 最主要的:
    这写是开源的, 并且很简单. 完全不用改变目前的任务业务逻辑.

do_email() 就是实际需要执行的任务. 这里模拟执行.

  • celery 任务启动

-A 指定到任务目录
-Q 是分配celery worker 的队列.

  • 测试 test.py

do_email.apply_async(queue=’hipri’) 这个指定了任务执行的队列.
并且把任务交给celery 来异步的调度.

观察任务执行的顺序, 能直观的发现任务是如何异步来执行的.

总结

上面是简单的例子. 在我们实际需求中, 值需要:

测试test.py 换乘tornado server, 在线接收任务.
就可以实时, 并发控制,可容错, 可视化的调度任务了.

  • scan 效率变慢, 并且经常超时. mapper 报OutOfOrderException. 影响任务执行效率.

所以有必要在理解scan 流程的基础上, 进行优化.
从应用hbase角度来讲, 需要理解scan 几个配置. 包括setCaching() setBatch(), 以及scan的过程.
